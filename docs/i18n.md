# i18n in This Repo

This portfolio uses build-time i18n for a static Vite multi-page site.

The goal is:
- keep a single source HTML template per page
- store copy in translation JSON files
- generate localized HTML before Vite builds `dist`
- preserve SEO by shipping already-localized HTML instead of translating in the browser at runtime

## How It Works

The flow is:
1. Source pages live in the normal repo locations like `index.html`, `sobre-mi/index.html`, and `casos/*.html`.
2. Those HTML files contain translation tokens like `{{ hero.title }}`.
3. `scripts/i18n/generate-localized-pages.mjs` reads the source HTML plus the translation files and generates localized HTML into `.i18n-build/`.
4. Vite builds from those generated pages.
5. `scripts/i18n/finalize-dist.mjs` moves the generated output into the final `dist/` route structure and also creates explicit default-locale aliases like `/es/`.

## Route Convention

The current route behavior is:
- `/page` uses the configured default language
- `/page/es` is the explicit Spanish route
- `/page/en` is the explicit English route

The default locale is configured in `i18n.config.js`.

## File Structure

Main files:
- `i18n.config.js`: locales, default locale, page registry, site origin
- `i18n.js`: locale helpers used by runtime code
- `scripts/i18n/generate-localized-pages.mjs`: build-time HTML localization
- `scripts/i18n/finalize-dist.mjs`: final `dist` route shaping
- `script.js`: locale-aware runtime behavior for shared UI like header, toggle, related cards, and form messages

Translations:
- `translations/es/shared.json`
- `translations/en/shared.json`
- `translations/es/runtime.json`
- `translations/en/runtime.json`
- `translations/es/case-studies.json`
- `translations/en/case-studies.json`
- `translations/<lang>/pages/...`

## Translation Convention

Use these namespaces:
- `shared`: copy reused across multiple pages at build time
- `runtime`: copy used by JavaScript-only UI behavior
- `case-studies`: shared case card metadata used in dynamic related cards and homepage case cards
- `pages/<page>.json`: page-specific content

Page files generally follow this structure:
- `meta`
- `hero`
- `metaSummary`
- `results`
- `situation`
- `task`
- `action`
- `result`
- `related`

Home uses page-specific sections such as:
- `hero`
- `intro`
- `featured`
- `experience`
- `approach`
- `contact`

## Token Convention in HTML

Source HTML should reference translation paths only.

Examples:

```html
<title>{{ meta.title }}</title>
<h1>{{ hero.title }}</h1>
<p>{{ intro.p1 }}</p>
```

For complex blocks, JSON can store arrays of HTML fragments. The build step joins them automatically.

Example:

```json
{
  "situation": {
    "contentHtml": [
      "<p>First paragraph.</p>",
      "<ul><li>Point one</li><li>Point two</li></ul>"
    ]
  }
}
```

Then in HTML:

```html
<div>{{ situation.contentHtml }}</div>
```

## Validation Rules

This repo now validates translation shape at build time.

The validator:
- uses the default locale as the schema reference
- fails the build if another locale is missing a key
- fails the build on type mismatches
- fails the build on empty strings
- fails the build when arrays do not match the reference length

The validator entrypoint is:
- `scripts/i18n/validate-translations.mjs`

This means translations should be kept structurally identical across locales.

## SEO Rules

Localization is resolved before deployment.

That means each generated localized page in `dist/` already includes:
- translated `<title>`
- translated `<meta name="description">`
- localized canonical URL
- `hreflang` alternate links
- localized HTML `lang`

Because Firebase serves the static `dist/` output, this is the correct SEO approach for this repo.

## Header and Language Toggle

The shared header is generated at build time and enhanced at runtime.

The language switcher:
- is rendered in the top-right header area
- links to explicit locale routes
- preserves the current route and hash at runtime

## Adding a New Page

When adding a new page:
1. Add the page to `i18n.config.js`.
2. Add translation keys under `translations/es/pages/...`.
3. Add `translations/en/pages/...` when English copy is ready.
4. Use tokens in the source HTML instead of literal copy.

## Adding a New Language

To add a new language:
1. Add the locale to `i18n.config.js`.
2. Add its locale metadata in `i18n.config.js`.
3. Create:
   - `translations/<locale>/shared.json`
   - `translations/<locale>/runtime.json`
   - `translations/<locale>/case-studies.json`
   - `translations/<locale>/pages/...`

## Important Repo-Specific Notes

- Keep source HTML as the single editable template. Do not create duplicate source HTML files per language.
- If a text is shared across pages, define it once in `shared` or another shared namespace instead of duplicating it.
- If SEO metadata should match page metadata, use the same translation key unless we explicitly decide otherwise.
- Internal links inside translation HTML fragments should prefer local routes like `/sobre-mi/` so the generator can localize them.
